WHAT IS PR SENTINEL?
--------------------
An autonomous agent that reviews GitHub Pull Requests the way a senior engineer
would. It runs real static analysis (ruff linter), reasons about logic and
security issues using an LLM, verifies its own claims against the actual code
before trusting them, and posts findings back as inline PR comments — running
live as a GitHub App, not a local demo.

TECH STACK
----------
- Orchestration    : LangGraph (StateGraph)
- LLM              : Groq (llama-3.3-70b-versatile)
- Linting          : ruff (via subprocess)
- Codebase Search  : ChromaDB (semantic search over repo)
- MCP Tools        : FastMCP
- API              : FastAPI
- Async Jobs       : Celery + Redis
- Database         : PostgreSQL
- Deployment       : Railway / Fly.io
- GitHub           : GitHub App (webhooks, JWT auth)

PHASE 1 — CORE AGENT LOGIC (COMPLETE)
================================================================================

STATUS: Done. Core loop working end-to-end and tested.

WHAT WAS BUILT:
---------------
1. ReviewState (TypedDict) — shared clipboard passed through all nodes:
   - raw_diff        : raw .patch text, input to the pipeline
   - parsed_files    : clean list of {filename, added_lines, removed_lines}
   - lint_issues     : findings from ruff (deterministic)
   - llm_findings    : findings from LLM (reasoning-based)
   - final_review    : merged, verified output

2. LangGraph pipeline — 5 nodes wired in sequence:
   ingest_diff → static_analysis → contextual_reasoning → aggregate_findings → format_review

3. parse_with_unidiff — converts raw diff text into structured Python objects
   using the `unidiff` library so downstream nodes work with clean data.

4. build_review_prompt — constructs the LLM instruction:
   - feeds only ADDED lines (not whole file, keeps prompt short and cheap)
   - asks for structured JSON output: {file, line, severity, comment, evidence}
   - explicitly says empty list is a valid, common outcome (prevents hallucination pressure)
   - few-shot examples of what a real finding looks like vs a non-issue
   - special guardrail for dependency/config files: only flag if syntax error,
     impossible constraint, or known CVE — not version range changes

5. call_llm — Groq API call:
   - model: llama-3.3-70b-versatile (upgraded from 8b-instant after catching
     reasoning failures — 8b was pattern-matching "version range = suspicious"
     without understanding what the range actually meant)
   - temperature: 0.2 (low randomness, consistent reviews)

6. parse_llm_response — defensive parser:
   - strips markdown code fences (```json ... ```) which models sometimes add
   - catches JSONDecodeError and returns [] instead of crashing
   - validates response is a list, not a dict or string

7. verify_findings — evidence-grounded hallucination check:
   - asks model to include "evidence" field: exact line from diff proving the issue
   - get_added_lines_text: flattens one file's added lines into a single searchable string
   - checks: evidence in file_lines → if not found, the finding is discarded
   - this is the key production pattern: never let model output go straight to
     user without grounding it in something verifiable

8. filter_for_posting — severity filter:
   - only pass medium/high findings to PR comments
   - log low-severity findings internally (don't spam PRs with nitpicks)

KEY DECISIONS MADE IN PHASE 1:
-------------------------------
- Model upgrade (8b → 70b) was higher-leverage than more prompt engineering
  for reasoning-quality issues.
- Evidence verification solves fabricated-claim hallucinations regardless of
  model quality. Both layers needed — they solve different failure modes.
- Tested against: synthetic diff (3 planted bugs), psf/requests PR #6169
  (dependency change — correctly returned []), real agent.py (3 real lint issues).

2.2 Map ruff severity properly

Map it to your scale:

  severity_map = {"error": "medium", "warning": "low"}

2.3 Implement filter_to_changed_lines
--------------------------------------
This is the most important step. Without it, ruff will flag every pre-existing
issue in the whole file, not just what this PR changed. That's unfair to the PR
author — not their mess to clean up.

  def filter_to_changed_lines(lint_issues, parsed_files):
      changed_lines_by_file = {
          f["filename"]: {line_no for line_no, _ in f["added_lines"]}
          for f in parsed_files
      }
      return [
          issue for issue in lint_issues
          if issue["line"] in changed_lines_by_file.get(issue["file"], set())
      ]

2.4 The file-content problem
------------------------------
ruff needs an actual .py file on disk — it can't lint a diff snippet in memory.
For local testing: save the "after" version of your test diff as a real .py file
and point run_linter at it.

In Phase 3 (GitHub integration), you will fetch full file content via the
GitHub API (get_file_content tool) and write it to a temp file before linting.
Design for this now: run_linter should accept a file_path argument, not assume
anything about where the file is.

  import tempfile

  def lint_from_content(filename: str, content: str) -> list:
      with tempfile.NamedTemporaryFile(suffix=".py", mode="w",
                                        delete=False) as tmp:
          tmp.write(content)
          tmp_path = tmp.name
      return run_linter(tmp_path, original_filename=filename)

2.5 Wire into static_analysis node
------------------------------------
  def static_analysis(state: ReviewState) -> ReviewState:
      all_issues = []
      for f in state["parsed_files"]:
          raw_issues = run_linter(f["filename"])
          filtered = filter_to_changed_lines(raw_issues, state["parsed_files"])
          all_issues.extend(filtered)
      state["lint_issues"] = all_issues
      return state

TOOLS TO BUILD (in app/mcp_tools/server.py):
---------------------------------------------

Tool 1: get_pr_diff(pr_id, repo)
  - Fetches raw .diff text from GitHub API for a given PR
  - Returns: raw diff string
  - Used by: ingest_diff node

Tool 2: get_file_content(path, ref, repo)
  - Fetches full file content at a given git ref (branch/commit)
  - Used by: lint node (needs real file, not just diff)
  - Also used by: search_codebase tool (to index repo content)

Tool 3: run_linter(filename, content)
  - Wraps ruff in a subprocess, returns structured findings
  - Accepts file content directly (writes to tempfile internally)
  - Returns: list of {file, line, severity, comment}

Tool 4: run_security_scan(filename, content)
  - Runs bandit (Python) or semgrep for security-specific patterns
  - Complements ruff (which is style/correctness focused, not security focused)
  - Returns: list of {file, line, severity, comment}
  - Install: pip install bandit

Tool 5: search_codebase(query, repo)
  - Embeds entire repo into ChromaDB (done once per repo, refreshed on push)
  - Semantic search to find related functions/classes beyond the diff
  - Example use: PR adds a function "calculate_discount" → agent searches codebase
    for other discount-related functions to check for inconsistency
  - This is your HybridRAG experience applied directly — dense + sparse retrieval
  - Returns: list of relevant code snippets with file + line references

Tool 6: post_review_comment(pr_id, repo, file, line, body)
  - Posts an inline comment on a specific line of the PR via GitHub API
  - Returns: comment URL (log for audit trail)

Tool 7: set_check_status(pr_id, repo, state, description)
  - Sets the green tick / red cross check on the PR
  - state: "success" | "failure" | "pending"
  - Call "pending" when review starts, "success"/"failure" when done

Tool 8: get_review_history(repo, limit)
  - Queries Postgres for past reviews on this repo
  - Used to provide context: "this file was reviewed 3 times before, common issue was X"
  - Turns your persistence layer into an active agent resource, not just logging

WHY A GITHUB APP AND NOT A PERSONAL TOKEN:
- A personal token only works for your own repos
- A GitHub App can be installed on ANY repo (yours, a friend's, a client's)
- "Installable on any repo" is the difference between "tool I made" and "product I shipped"
- GitHub Apps use short-lived JWT tokens (auto-refreshed) — more secure than a
  static token sitting in your .env forever

3.2 Webhook endpoint (app/github/webhook.py)
---------------------------------------------
  from fastapi import APIRouter, Request, HTTPException
  import hmac, hashlib

  router = APIRouter()

  @router.post("/webhook")
  async def handle_webhook(request: Request):
      # Step 1: verify the request actually came from GitHub
      payload = await request.body()
      signature = request.headers.get("X-Hub-Signature-256", "")
      secret = os.environ["GITHUB_WEBHOOK_SECRET"].encode()
      expected = "sha256=" + hmac.new(secret, payload, hashlib.sha256).hexdigest()
      if not hmac.compare_digest(expected, signature):
          raise HTTPException(status_code=401, detail="Invalid signature")

      # Step 2: parse event type and route
      event = request.headers.get("X-GitHub-Event")
      body = await request.json()

      if event == "pull_request" and body["action"] in ("opened", "synchronize"):
          pr_id = body["pull_request"]["number"]
          repo = body["repository"]["full_name"]
          # enqueue async job (Phase 4) — don't block the webhook response
          review_pr.delay(pr_id, repo)

      return {"status": "ok"}

HMAC SIGNATURE VERIFICATION IS MANDATORY:
Without it, anyone can send fake events to your webhook endpoint.
Always use hmac.compare_digest (not ==) to prevent timing attacks.
Mention this in your README and interviews — it shows production security awareness.

3.3 GitHub App auth (app/github/auth.py)
-----------------------------------------
GitHub Apps authenticate with short-lived JWT tokens, not static API keys.
Flow:
  1. Sign a JWT using your App's private key + App ID (expires in 10 min)
  2. Use that JWT to request an Installation Access Token for a specific repo
  3. Use the Installation Access Token for all actual API calls
  4. Refresh when it expires (they last 1 hour)

  import jwt, time, requests

  def get_installation_token(installation_id):
      payload = {
          "iat": int(time.time()),
          "exp": int(time.time()) + 600,
          "iss": os.environ["GITHUB_APP_ID"]
      }
      private_key = open("github_private_key.pem").read()
      jwt_token = jwt.encode(payload, private_key, algorithm="RS256")

      resp = requests.post(
          f"https://api.github.com/app/installations/{installation_id}/access_tokens",
          headers={"Authorization": f"Bearer {jwt_token}",
                   "Accept": "application/vnd.github+json"}
      )
      return resp.json()["token"]

3.4 GitHub API client (app/github/client.py)
---------------------------------------------
Wrap the common API calls your MCP tools need:

  def get_pr_diff(pr_id, repo, token):
      resp = requests.get(
          f"https://api.github.com/repos/{repo}/pulls/{pr_id}",
          headers={"Authorization": f"Bearer {token}",
                   "Accept": "application/vnd.github.v3.diff"}
      )
      return resp.text

  def post_review_comment(pr_id, repo, token, commit_sha, path, line, body):
      requests.post(
          f"https://api.github.com/repos/{repo}/pulls/{pr_id}/comments",
          headers={"Authorization": f"Bearer {token}"},
          json={"body": body, "commit_id": commit_sha,
                "path": path, "line": line}
      )

  def set_check_status(repo, token, sha, state, description):
      requests.post(
          f"https://api.github.com/repos/{repo}/statuses/{sha}",
          headers={"Authorization": f"Bearer {token}"},
          json={"state": state, "description": description,
                "context": "PR Sentinel"}
      )

PHASE 4 — ASYNC JOB PROCESSING
================================================================================

STATUS: Not started.

GOAL: A PR review takes 30 seconds to 2 minutes (LLM calls, linting, API calls).
GitHub expects your webhook to respond with HTTP 200 within ~10 seconds or it
marks the delivery as failed. So you can't do the review inline — you respond
immediately and process in the background.

4.1 Install Celery + Redis
---------------------------
  pip install celery redis

Redis acts as the message broker: the webhook puts a job on the queue, the
Celery worker picks it up and runs the actual review.

4.2 Define the task (app/workers/tasks.py)
-------------------------------------------
  from celery import Celery

  celery_app = Celery("pr_sentinel",
                       broker=os.environ["REDIS_URL"],
                       backend=os.environ["REDIS_URL"])

  @celery_app.task(bind=True, max_retries=3, default_retry_delay=30)
  def review_pr(self, pr_id: int, repo: str):
      try:
          token = get_installation_token(get_installation_id(repo))
          raw_diff = get_pr_diff(pr_id, repo, token)

          # run your LangGraph agent
          result = graph.invoke({"raw_diff": raw_diff})

          # post findings back
          for finding in result["final_review"]:
              post_review_comment(pr_id, repo, token, ...)
          set_check_status(repo, token, sha, "success", "Review complete")

          # persist to Postgres
          save_review(pr_id, repo, result["final_review"])

      except Exception as exc:
          set_check_status(repo, token, sha, "failure", "Review failed")
          raise self.retry(exc=exc)

KEY POINTS:
- max_retries=3: if the LLM call or a GitHub API call fails transiently, retry
  automatically up to 3 times — don't just silently drop the review
- Always set check status to "failure" if something goes wrong so the PR author
  knows the review didn't complete, rather than hanging on "pending" forever
- bind=True: lets the task access self for retry logic

4.3 Webhook just enqueues, doesn't wait
-----------------------------------------

  review_pr.delay(pr_id, repo)   # returns immediately
  return {"status": "ok"}        # GitHub gets its 200 right away

5.1 Models (app/db/models.py)
-------------------------------
  from sqlalchemy import Column, Integer, String, JSON, DateTime
  from sqlalchemy.ext.declarative import declarative_base
  Base = declarative_base()

  class Review(Base):
      __tablename__ = "reviews"
      id          = Column(Integer, primary_key=True)
      pr_id       = Column(Integer)
      repo        = Column(String)
      verdict     = Column(String)           # "issues_found" / "clean"
      findings    = Column(JSON)             # full list of findings
      reviewed_at = Column(DateTime)
      model_used  = Column(String)           # track which model produced this

  class Finding(Base):
      __tablename__ = "findings"
      id          = Column(Integer, primary_key=True)
      review_id   = Column(Integer)
      file        = Column(String)
      line        = Column(Integer)
      severity    = Column(String)
      comment     = Column(String)
      evidence    = Column(String)
      was_valid   = Column(Boolean, nullable=True)   # for future human feedback

5.2 Why track model_used and was_valid?
-----------------------------------------
model_used: when you upgrade models in future (or do A/B testing between
  model versions), you can query "which model had fewer false positives"
  against real data rather than guessing.

was_valid: optional field for future where a human can mark findings as
  genuine or false positives. Over time this becomes a feedback dataset.
  Not needed at launch, but designing the column in now costs nothing.

================================================================================
COMMON PITFALLS TO AVOID
================================================================================

1. NEVER trust LLM output without verification
   Always run findings through verify_findings before posting to a PR.
   A hallucinated "SQL injection" comment on clean code will get your app
   uninstalled immediately.

2. NEVER skip HMAC verification on the webhook
   Without it, anyone can POST fake events to your endpoint.

3. NEVER use a personal access token for GitHub auth
   Use the GitHub App JWT flow. Personal tokens only work on your own repos
   and are a security risk if leaked.

4. DON'T flag pre-existing lint issues
   Use filter_to_changed_lines so you only comment on what this PR introduced,
   not problems that existed before the author touched the file.

5. DON'T block the webhook response
   Use Celery to process the review asynchronously. GitHub will mark your
   webhook as failed if you don't respond within ~10 seconds.

6. DON'T deploy to Hugging Face Spaces
   It doesn't support always-on multi-process Docker deploys.
   Use Railway or Fly.io.
